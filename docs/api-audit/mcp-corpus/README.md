# Axxon One MCP Corpus

This folder is the Phase 0 structured data snapshot for the planned public Axxon One MCP server.

Generated by:

```bash
/tmp/axxon-grpc-venv/bin/python arm64-docker/tools/generate_mcp_corpus.py
```

Files:

- `api_methods.json`: 361 gRPC methods with package, service, request/response names, streaming mode, safety class, HTTP annotation, proto path, and live status.
- `http_endpoints.json`: 221 annotated `/v1/...` HTTP endpoints parsed from the proto-derived endpoint catalog.
- `task_recipes.json`: task-oriented sections from the integration playbook plus mutation playbook references.
- `fixtures.json`: coverage counts and the six current fixture-needed rows (five legacy fixture gaps plus TFA mutations).
- `safety_policies.json`: default read-only posture, bounded-stream limits, mutation approval gates, rollback requirements, and redaction rules.
- `known_behaviors.json`: verified demo-stand quirks and compatibility notes that an MCP server should not rediscover from chat history.

This corpus is sanitized source material for MCP implementation. It contains no credentials or bearer tokens.

## Phase 1 Docs Query Layer

The first docs-only MCP layer is implemented as:

```text
arm64-docker/tools/axxon_mcp_docs.py
arm64-docker/tools/axxon_mcp_server.py
arm64-docker/tools/requirements-mcp.txt
```

It does not connect to an Axxon server and requires no credentials. The query module and FastMCP server expose the same operations planned for the MCP tools:

- `search_api_docs(query)`
- `get_api_method(fqmn)`
- `get_http_endpoint(path_or_topic)`
- `get_verified_example(topic)`
- `explain_task_recipe(task)`
- `list_remaining_gaps()`


## Phase 2 Read-Only Live Inspection

The first live-inspection layer is implemented as:

```text
arm64-docker/tools/axxon_mcp_live.py
```

It reuses `AxxonApiClient`, keeps credentials in memory, and returns only summarized/redacted inventory data. Live tools are not enabled by default. Start the MCP server with `--enable-live` only when the target profile is provided through environment variables:

```bash
AXXON_USERNAME=root \
AXXON_PASSWORD='<password>' \
/tmp/axxon-grpc-venv/bin/python arm64-docker/tools/axxon_mcp_server.py --enable-live --transport stdio
```

Additional MCP tools enabled in live mode:

- `connect_axxon_profile(profile)`
- `list_cameras(filter_text, limit)`
- `list_archives(filter_text, limit)`
- `list_config_units(filter_text, limit)`
- `list_detectors(camera_or_host, limit)`
- `list_appdata_detectors(camera_or_host, limit)`
- `find_event_suppliers(camera_or_detector, limit)`
- `find_metadata_endpoints(camera_or_detector, limit)`
- `preflight_task(task)`
- `get_archive_intervals(camera, hours, max_count, min_gap_ms)`
- `subscribe_events_bounded(subjects, event_types, timeout, limit)`

`get_archive_intervals` wraps `ArchiveService.GetHistory2` and clamps `hours` to 24 and `max_count` to 200. `subscribe_events_bounded` wraps `DomainNotifier.PullEvents` and clamps `timeout` to 30 seconds and `limit` to 500 events. Both tools require a camera/subject from the connected inventory.

Event and metadata search tools (live-verified 2026-05-15 in `mcp-feature-expansion-2026-05-15-run.log`):

- `list_event_types()` тАФ returns the full `EEventType` enum (67 values on demo) with both name and numeric value.
- `list_detector_kinds()` тАФ returns documented detector kinds (AVDetector: MotionDetection / SceneDescription / NeuroTracker; AppDataDetector: MoveInZone / OneLineCrossing / LongInZone / LostObject / AbandonedObject) unioned with any live `enum_constraint` values discovered on existing detector units.
- `search_events(subjects, event_types, hours, limit, descending)` wraps `EventHistoryService.ReadEvents`. Clamps `hours` to [0.05, 168.0] and `limit` to [1, 1000]. `event_types` accepts either `ET_*` enum names or numeric values.
- `pull_metadata_bounded(access_point, timeout, limit)` wraps `MetadataService.PullMetadata` (bidirectional stream). Clamps `timeout` to [1.0, 30.0]s and `limit` to [1, 500] frames. Requires an `AVDetector.*/SourceEndpoint.vmda` access point (or any metadata source endpoint).

## Phase 3 Controlled Operator Tools

The first operator layer is implemented as:

```text
arm64-docker/tools/axxon_mcp_operator.py
```

Operator mutations are off by default. Start the MCP server with `--enable-operator` AND set `AXXON_OPERATOR_APPROVE=1` to enable them. Workflows expose four MCP tools each:

- `list_operator_workflows()`
- `plan_operator_workflow(workflow, params)` тАФ typed plan, no server call
- `apply_operator_plan(plan_id, confirmation)` тАФ requires matching confirmation token
- `verify_operator_plan(plan_id)` тАФ proves created objects are still present
- `rollback_operator_plan(plan_id, confirmation)` тАФ removes every UID created by apply

The `axxon://operator/audit-log` resource returns the in-memory audit trail for the session. Each plan stores a typed `ChangeConfig` payload and a rollback strategy that records every UID created by `apply` so `rollback` removes exactly that set.

Current workflow set:

Live-verified end-to-end on the demo stand 2026-05-14:

- `temp_camera` тАФ adds a Virtual/several-streams `DeviceIpint` and removes it.
- `temp_archive` тАФ adds a Gray object-storage `MultimediaStorage` (`day_depth=0`) and removes it.
- `temp_av_detector` тАФ adds an `AVDetector` (default `MotionDetection`) bound to a caller-supplied `video_source_ap` and removes it. Returns `status: gap` if `video_source_ap` is missing.

Also live-verified on the demo stand 2026-05-14:

- `temp_appdata_detector` тАФ chained two-step workflow when no `vmda_source_ap` is supplied: creates a SceneDescription AVDetector first, waits for its `SourceEndpoint.vmda` to publish, then creates the AppDataDetector (default `MoveInZone`) against it. Rollback removes both in reverse order. Returns `status: gap` only if `video_source_ap` is missing.
- `temp_device_template` тАФ uses `ChangeTemplates` to create a device template attached to a caller-supplied `camera_uid`, then removes it. Returns `status: gap` if `camera_uid` is missing.
- `external_event_inject` тАФ POSTs a one-shot event to `/v1/detectors/external:raiseOccasionalEvent` against a caller-supplied `access_point` (an existing `DetectorEx.*/EventSupplier`). One-shot: rollback is a no-op.
- `temp_macro` тАФ creates a disabled, empty `MacroConfig` via `LogicService.ChangeMacros`, then removes it. Self-contained: no fixture required.

Persistent workflows (no auto-rollback; caller owns lifecycle; rollback path still works on demand):

- `create_camera(display_name, vendor?, model?, display_id?, ip?, login?, password?)` тАФ persistent `DeviceIpint` via `ChangeConfig`. Defaults to a Virtual several-streams device. Live-verified 2026-05-15.
- `create_macro(name, enabled?, conditions?, actions?, guid?)` тАФ persistent `MacroConfig` via `LogicService.ChangeMacros`. Client-generated GUID by default. Live-verified 2026-05-15.
- `create_layout(name, cells, rows?, cols?, layout_id?)` тАФ persistent layout via `LayoutManager.Update.created`. `cells` is a list of camera access-point strings (placed row-major) or per-cell `{access_point, position, width, height}` dicts. Live-verified 2026-05-15.
- `set_unit_properties(uid, properties)` тАФ generic property edit on an existing unit via `ChangeConfig.changed[]`. Covers detector parameter changes after creation. Live-verified 2026-05-15.

See `arm64-docker/docs/api-audit/mcp-operator-smoke-latest.md` and `arm64-docker/docs/api-audit/mcp-feature-expansion-2026-05-15-run.log` for evidence.
