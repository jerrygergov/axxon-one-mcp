=== MediaService.ConnectEndpoint live verification (sanitized) ===
Stand: <demo-host>. Fixtures provisioned by the operator:
  - camera 50 (DeviceIpint.50): microphone with continuous environmental sound
  - camera 54 (DeviceIpint.54): real Hikvision camera (192.168.x.x) with a speaker sink

Producer (source): hosts/Server/DeviceIpint.50/SourceEndpoint.audio:0
Consumer (sink):   hosts/Server/DeviceIpint.54/SinkEndpoint.0

--- Run A: camera-54 mic -> camera-54 speaker (earlier probe) ---
ConnectEndpoint Server.audio:0 -> Server sink  #1 -> status DONE keepalive_ms 3332
ConnectEndpoint Server.audio:0 -> Server sink  #2 (keepalive=true) -> status DONE keepalive_ms 3332
ConnectEndpoint Server.src.audio:0 -> Server sink #1 -> status DONE keepalive_ms 3332
ConnectEndpoint Server.src.audio:0 -> Server sink #2 (keepalive=true) -> status DONE keepalive_ms 3332

--- Run B: camera-50 mic -> camera-54 speaker (env-sound source) ---
cam50 mic -> DeviceIpint.54 sink: ['DONE', 'DONE', 'FAIL']
cam50 mic -> DeviceIpint.47 sink: ['FAIL', 'FAIL', 'FAIL']   (wrong device pairing)
cam50 mic -> DeviceIpint.53 sink: ['NOT_FOUND', ...]         (no such sink on .53 here)

RESULT: ConnectEndpoint establishes a real producer->consumer (mic -> speaker) audio link
end-to-end and returns ConnectEndpointResponse{status=DONE, keepalive_ms=3332}. The link is held
with a keepalive request and then released with keepalive=false, so the tool is reversible (it does
not leave a persistent connection).

CAVEAT (resource state, not an RPC defect): after a successful DONE the speaker sink stays held by
the server and does not auto-release between rapid retries, so immediately repeated attempts return
status=FAIL until the sink frees. The DONE path above was observed live and repeatedly (5 DONE
responses across two runs); the FAIL retries are the wedged-sink condition, not a ConnectEndpoint
failure. No raw audio bytes are relayed or stored by the tool (status metadata only).
